# EscapeHint PRD 작성을 위한 필요 정보 정리

## 현재 상황

작성된 문서:

- ✅ `docs/1-domain-definition.md` - 도메인 정의서 (완성)

목표:

- **PRD(Product Requirements Document)** 작성

## PRD란?

PRD는 "무엇을(What)", "왜(Why)"에 집중하는 문서입니다.

- 도메인 정의서: 비즈니스 도메인의 개념, 규칙, 엔티티 정의
- PRD: 제품의 목적, 사용자 요구사항, 기능 명세, 성공 지표

## 도메인 정의서에 이미 있는 정보

✅ **사용 가능한 정보**:

1. 서비스 개요 (1장)

   - 비즈니스 목적
   - 핵심 가치
   - 사용자 유형 (플레이어/관리자)

2. 주요 유스케이스 (3장)

   - UC-01~UC-09: 상세한 사용자 시나리오
   - 각 유스케이스별 목적, 절차, 결과

3. 도메인 규칙 (4장)

   - 비즈니스 규칙 (BR-01~BR-09)
   - 데이터 검증 규칙 (VR-01~VR-03)

4. 품질 속성 메트릭 (5.6장)
   - 성능, 사용성, 신뢰성 지표

## PRD 작성을 위해 추가로 필요한 정보

### 카테고리 1: 제품 전략 및 비전 (Product Strategy)

#### 1.1 제품 목표 및 비전

**도메인 정의서에 없는 정보**:

- **제품 비전**: 1-3년 후 이 제품이 달성하고자 하는 궁극적 목표
- **핵심 문제 정의**: 현재 방탈출 카페들이 겪는 구체적 페인 포인트
  - 예: "종이 힌트는 분실 위험, 직원 호출은 몰입도 저하"
- **경쟁 제품 분석**: 유사한 힌트 제공 시스템이 있다면 차별점
- **성공 가설**: "만약 X를 제공하면 Y한 결과가 나올 것"

#### 1.2 타겟 사용자 페르소나

**도메인 정의서에 있는 것**: 플레이어/관리자 두 유형
**추가 필요 정보**:

- **플레이어 페르소나**:

  - 연령대, 방탈출 경험 수준 (초보/중급/고급)
  - 게임 중 기대하는 경험 (몰입감, 성취감 등)
  - 힌트에 대한 태도 (빨리 보고 싶음 vs 최대한 안 보려 함)
  - 디지털 리터러시 (스마트폰 사용 편의성)

- **관리자 페르소나**:
  - 역할 (카페 사장님, 직원, 테마 디자이너)
  - IT 숙련도 (비전문가 가정)
  - 하루 평균 관리 시간 (5분 이내 목표?)
  - 주요 불편사항 (종이 힌트 정리, 힌트 변경 어려움 등)

#### 1.3 시장 분석 및 비즈니스 모델

**추가 필요 정보**:

- **타겟 시장 규모**: 국내 방탈출 카페 수, 평균 테마 수
- **수익 모델**:
  - 무료 오픈소스?
  - 라이선스 판매? (카페당 월/년 구독)
  - 프리미엄 기능? (통계 분석, 다국어 등)
- **도입 장벽**: 방탈출 카페가 채택하기 위한 조건
  - 예: 태블릿 구비, 인터넷 연결, 초기 설정 시간

### 카테고리 2: 기능 우선순위 및 범위 (Feature Scope)

#### 2.1 MVP(Minimum Viable Product) 정의

**추가 필요 정보**:

- **1차 출시 필수 기능**:

  - 도메인 정의서의 UC-01~09 중 어디까지?
  - 예: 통계 기능(UC-08)은 2차 버전으로 미룰 수 있음

- **제외할 기능**:
  - 이미지/비디오 힌트 (향후 확장)
  - 다국어 지원 (향후 확장)
  - 여러 방 동시 관리 (향후 확장)

#### 2.2 기능별 우선순위

**제안 필요**:

- **P0 (필수)**: 힌트 조회, 타이머, 테마/힌트 CRUD
- **P1 (중요)**: 진행률 표시, 세션 모니터링
- **P2 (선택)**: 통계 분석, 고급 관리자 기능

#### 2.3 사용자 스토리

**도메인 정의서에 있는 것**: 유스케이스 (시스템 관점)
**추가 필요**: 사용자 관점 스토리

- "플레이어로서, 힌트 코드를 빠르게 입력하고 싶다. 왜냐하면 게임에 집중하고 싶기 때문이다."
- "관리자로서, 힌트를 5분 안에 등록하고 싶다. 왜냐하면 바쁜 운영 시간에 빠르게 처리해야 하기 때문이다."

### 카테고리 3: 사용자 경험 요구사항 (UX Requirements)

#### 3.1 플레이어 화면 UX 원칙

**도메인 정의서에 있는 것**: "큰 글자, 직관적 버튼, 몰입 방해 최소화"
**구체화 필요**:

- **글자 크기 기준**: 최소 24px? 어두운 방에서도 읽기 쉬운가?
- **색상 전략**: 테마별 커스터마이징? 고정 다크 모드?
- **입력 방식**:
  - 물리 키보드 vs 화면 키보드?
  - 음성 입력 지원?
  - 바코드/QR 코드 스캔?
- **피드백 디자인**:
  - 정확한 코드 입력 시: 효과음? 진동?
  - 잘못된 코드 입력 시: 어떤 메시지?

#### 3.2 관리자 화면 UX 원칙

**구체화 필요**:

- **힌트 순서 조정 방법**: 드래그앤드롭? 번호 입력?
- **대량 등록 지원**: 엑셀 업로드? CSV 가져오기?
- **미리보기 기능**: 플레이어 화면으로 미리보기?

#### 3.3 디바이스 및 환경

**추가 필요 정보**:

- **주요 디바이스**:
  - 태블릿 (10인치 기준?)
  - 스마트폰 (플레이어 개인 폰?)
  - PC (관리자용?)
- **환경 조건**:
  - 어두운 방 (백라이트 최대, 고대비)
  - 소음 환경 (음향 피드백 제한적)
  - 터치 장갑 착용 가능성

### 카테고리 4: 비기능 요구사항 명세 (Non-Functional Requirements)

#### 4.1 성능 요구사항 구체화

**도메인 정의서에 있는 것**:

- API 응답 P95 < 200ms
- 타이머 정확도 ±500ms/60초
- 동시 세션 20개

**구체화 필요**:

- **최소 네트워크 환경**: 3G 환경에서도 작동?
- **오프라인 지원**: 힌트 데이터 로컬 캐싱?
- **로딩 시간**: 초기 로딩 3초 이내?

#### 4.2 보안 요구사항

**도메인 정의서에 있는 것**: HTTPS, XSS 방지, 관리자 인증
**구체화 필요**:

- **데이터 암호화**: DB에 정답 평문 저장? 암호화?
- **세션 보안**: 세션 하이재킹 방지 전략?
- **관리자 인증 수준**:
  - 2단계 인증 필요?
  - IP 화이트리스트?
  - 세션 타임아웃 (2시간?)

#### 4.3 접근성 (Accessibility)

**추가 필요 정보**:

- **시각 장애 대응**: 스크린 리더 지원?
- **색맹 대응**: 색상만으로 정보 전달 금지?
- **손 떨림 대응**: 터치 영역 충분히 큼? (최소 44x44px?)

### 카테고리 5: 운영 및 지원 요구사항 (Operations)

#### 5.1 배포 및 설치

**추가 필요 정보**:

- **배포 환경**:
  - 클라우드 SaaS? (카페가 접속만)
  - 온프레미스? (카페가 직접 서버 설치)
  - 하이브리드? (로컬 네트워크 + 클라우드 백업)
- **설치 복잡도**: 비IT 사장님이 30분 안에 설치 가능?
- **초기 설정**: 데모 데이터 제공? 튜토리얼?

#### 5.2 유지보수 및 업데이트

**추가 필요 정보**:

- **업데이트 방식**: 자동? 수동?
- **데이터 백업**: 자동 백업? 주기?
- **장애 대응**: 24시간 지원? 커뮤니티 지원?

#### 5.3 교육 및 문서화

**추가 필요 정보**:

- **사용자 매뉴얼**:
  - 플레이어용: 1페이지 가이드?
  - 관리자용: 비디오 튜토리얼?
- **온보딩**: 첫 테마 등록까지 단계별 가이드?

### 카테고리 6: 성공 지표 (Success Metrics)

#### 6.1 정량적 지표 (Quantitative Metrics)

**추가 필요 정보**:

- **채택률**:
  - 3개월 내 X개 카페 도입 목표?
  - 월간 활성 사용자(MAU)?
- **사용성 지표**:
  - 힌트 조회 성공률 > 95%?
  - 관리자 힌트 등록 완료율 > 90%?
  - 평균 세션 에러율 < 1%?
- **성능 지표**:
  - 서버 가동률 > 99.5%?
  - 평균 응답 시간 < 150ms?

#### 6.2 정성적 지표 (Qualitative Metrics)

**추가 필요 정보**:

- **사용자 만족도**:
  - NPS(Net Promoter Score) 목표치?
  - 플레이어 만족도: "힌트 시스템이 게임 경험을 방해하지 않았나?" 설문
  - 관리자 만족도: "종이 힌트 대비 얼마나 편한가?" 설문

#### 6.3 비즈니스 지표

**추가 필요 정보**:

- **ROI**: 카페 입장에서 도입 비용 대비 효과
  - 직원 시간 절약 (시간당 X회 힌트 제공 불필요)
  - 고객 만족도 향상 → 재방문율 증가?
- **확장성**: 테마당 평균 힌트 수, 카페당 평균 테마 수

### 카테고리 7: 제약사항 및 가정 (Constraints & Assumptions)

#### 7.1 기술적 제약사항

**추가 필요 정보**:

- **브라우저 지원**: 최신 Chrome/Safari만? IE 11 지원?
- **최소 디바이스 사양**: iOS 12+, Android 8+?
- **네트워크 의존성**: 완전 오프라인 불가?

#### 7.2 비즈니스 제약사항

**추가 필요 정보**:

- **예산**: 개발 예산, 운영 예산
- **일정**: 1차 출시 목표일?
- **인력**: 개발팀 규모 (1인 vs 팀)?

#### 7.3 가정(Assumptions)

**명시 필요**:

- "카페는 안정적인 Wi-Fi를 제공한다"
- "플레이어는 스마트폰 또는 제공된 태블릿을 사용한다"
- "관리자는 기본적인 웹 브라우저 사용법을 안다"
- "하나의 카페는 동시에 최대 5개 세션을 운영한다"

### 카테고리 8: 위험 관리 (Risk Management)

#### 8.1 기술적 위험

**추가 필요 정보**:

- **타이머 정확도**: 브라우저 백그라운드에서 부정확 → 서버 동기화로 해결
- **네트워크 단절**: 힌트 조회 실패 → 로컬 캐싱으로 완화
- **동시 접속**: 예상보다 많은 세션 → 로드 밸런싱 필요?

#### 8.2 사용자 경험 위험

**추가 필요 정보**:

- **관리자 모드 노출**: 플레이어가 우연히 발견 → 강력한 보안 필요
- **힌트 코드 유출**: SNS에 정답 공유 → 코드 주기적 변경 기능?
- **디바이스 고장**: 게임 중 태블릿 꺼짐 → 세션 복구 기능 필수

### 카테고리 9: 향후 로드맵 (Future Roadmap)

#### 9.1 2차 버전 후보 기능

**도메인 정의서 5.5에서 언급**:

- 다국어 지원 (힌트 텍스트)
- 이미지/오디오/비디오 힌트
- 통계 및 분석 (테마별 클리어율, 평균 힌트 사용 수)
- 여러 방 동시 관리

#### 9.2 장기 비전

**추가 고려**:

- AI 기반 힌트 추천: 플레이 패턴 분석 → 적절한 타이밍에 힌트 제안
- 소셜 기능: 플레이 기록 공유, 리더보드
- 프랜차이즈 지원: 여러 지점 통합 관리

## 요약: PRD 작성 시 수집/결정해야 할 핵심 질문

### 전략적 질문

1. 이 제품의 1년 후 목표는? (도입 카페 수, 사용자 만족도)
2. 경쟁 제품이 있는가? 차별점은?
3. 수익 모델은? (오픈소스 vs 유료 라이선스)
4. MVP 범위는? (통계 기능 포함? 이미지 힌트 포함?)

### 사용자 경험 질문

5. 플레이어의 주요 디바이스는? (개인 폰 vs 제공 태블릿)
6. 관리자 모드 진입 방법은? (숨겨진 버튼 vs URL+비밀번호)
7. 힌트 코드 입력 방식은? (키보드 vs QR 코드 스캔)
8. 어두운 방에서 가독성 확보 방법은? (글자 크기, 색상 대비)

### 기술적 질문

9. 배포 방식은? (클라우드 SaaS vs 온프레미스)
10. 오프라인 지원 필요? (로컬 캐싱 수준)
11. 지원 브라우저/OS 범위는?
12. 데이터 백업 전략은?

### 운영적 질문

13. 출시 목표일은?
14. 초기 설정 복잡도 허용 범위는? (30분? 1시간?)
15. 사용자 지원 방식은? (문서만? 실시간 채팅?)

## PRD 구조 및 도메인 정의서 매핑

PRD를 작성할 때 다음과 같이 구성하며, 도메인 정의서에서 재사용 가능한 부분을 표시:

### PRD 섹션별 구성

| PRD 섹션                           | 도메인 정의서 활용         | 추가 필요 정보                                     |
| ---------------------------------- | -------------------------- | -------------------------------------------------- |
| **1. Executive Summary**           | 1.1 비즈니스 목적 (일부)   | 제품 비전, 경쟁사 분석, MVP 범위                   |
| **2. Product Overview**            | 1.2 핵심 가치              | 시장 분석, 수익 모델                               |
| **3. User Personas**               | 1.3 사용자 유형 (기본)     | 페르소나 상세화 (연령, 경험, 니즈)                 |
| **4. User Stories**                | 3장 유스케이스 (변환 필요) | As a [role], I want [goal], so that [benefit] 형식 |
